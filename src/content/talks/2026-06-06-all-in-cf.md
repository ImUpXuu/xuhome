        用cloudflare反代了半年vercel，我终于切到了纯cf
有些朋友可能不知.[](https://upxuu.com)一直都是使用vercel部署的
你可能很疑惑，那为啥我ping过去是cf的anycast呢but...当你去看请求头，哎为啥还有代表命中vercel缓存的呢？？？，事实上，当时因为懒得做pages优选，因此一致是cf反代vc/（pages优选主域要dns托管到cloudflare外，十分的麻烦啊）
但是 最近发现workers竟然可以和pages一样托管静态资源 这里原理我们直接ds吧
当你 wrangler deploy 后，Cloudflare 内部做了两件事：
1. 静态文件 → 内容寻址存储
dist/index.html
dist/about/index.html  
dist/文章A/index.html
wrangler 逐个计算 blake3 哈希 → 上传到 Cloudflare 的内容寻址存储 → 边缘节点 CDN 直出。匹配到的请求根本不碰 Worker 运行时，性能等同 Cloudflare Pages。
2. Worker 代码 → V8 Isolate
src/worker.js → 编译 → 部署到 V8 isolate
Worker 只处理 CDN 没匹配到的请求。env.ASSETS 是一个绑定的存储桶句柄，调用 .fetch(req) 就是去查那个内容寻址存储。
所以分层结构是：
请求 → Cloudflare 边缘节点
  ├─ 请求路径匹配 dist/ 里的文件 → CDN 直接返回（零冷启动）
  └─ 不匹配任何文件 → 转发给 V8 isolate 跑 worker.js
       └─ env.ASSETS.fetch(req) → 再去查一次存储
            ├─ 找到 → 返回文件
            └─ 还是没找到 → worker 可以返回 404、SPA fallback 等
main 必须指定的原因不是它每次都跑，而是wrangler 的命令模型强制要求——它先看你是 Worker 项目还是纯静态，main 就决定了 "这是 Worker 项目"，然后 [assets] 告诉它 "顺便把 dist/ 挂上去"。没有 main，wrangler 就不知道要部署什么类型的项目。
总之 最后我换成了worker 感觉快了些吧
